# Исправление нарушения Dependency Inversion Principle (DIP)

## Описание проекта
Данный проект выполнен в рамках учебного задания и демонстрирует исправление нарушения
принципа инверсии зависимостей (**Dependency Inversion Principle, DIP**) из набора принципов SOLID.

В задании были приведены примеры кода на языках Java и C#, в которых высокоуровневые модули
напрямую зависели от конкретных реализаций низкоуровневых классов.  
В данном проекте показано корректное архитектурное решение этих примеров на языке Java.

## Цель работы
Целью проекта является:
- выявление нарушения принципа DIP;
- демонстрация способа его исправления;
- применение абстракций и внедрения зависимостей;
- получение гибкой и расширяемой архитектуры.

## Суть нарушения DIP
В исходных примерах:
- высокоуровневые классы (`DataProcessor`, `NotificationManager`);
- напрямую создавали объекты конкретных классов (`MySQLDatabase`, `EmailService`);
- использовали оператор `new` внутри бизнес-логики.

Это приводило к жёсткой связности и нарушению принципа DIP, согласно которому
модули высокого уровня не должны зависеть от модулей низкого уровня, а оба должны
зависеть от абстракций.

## Принятое решение
Для исправления нарушения DIP были выполнены следующие шаги:
1. Введены абстракции в виде интерфейсов (`Database`, `NotificationService`).
2. Конкретные реализации вынесены в отдельные классы (`MySQLDatabase`, `PostgreSQLDatabase`, `EmailService`).
3. Высокоуровневые модули получают зависимости извне через конструктор
   (Dependency Injection).
4. Выбор конкретной реализации осуществляется в точке входа приложения.

## Используемые паттерны проектирования

### Dependency Injection
Используется для передачи зависимостей извне, что устраняет жёсткую зависимость
высокоуровневых классов от конкретных реализаций.

### Strategy (потенциально)
Использование интерфейсов позволяет в дальнейшем добавлять новые реализации
без изменения существующей бизнес-логики.
